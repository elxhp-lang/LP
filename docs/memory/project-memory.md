# 跨平台插件化 AI 应用项目记忆库（主文档）

## 文档目的
- 本文档是本项目的长期记忆基线，用于保证跨会话、跨成员、跨阶段开发的一致性。
- 后续所有架构、代码、文档与交付物，默认先对齐本文档再执行。

## 项目定位
- 构建跨平台插件化 AI 应用，覆盖 Web 端与移动端。
- 核心架构采用“轻量核心平台 + 可插拔插件生态”。
- MVP 首期聚焦跨境电商场景，后续可扩展至建筑、母婴、教育等行业。

## 核心目标（新增基线）
- 解耦架构：核心基座仅负责用户认证、RBAC 权限、插件生命周期管理、AI 模型调度等基础能力，所有业务功能通过插件实现。
- 跨端一致性：Web 端与移动端共享业务逻辑，仅在 UI 交互层做差异化适配。
- 生态扩展：内置插件市场，支持开发者上架、定价与收益分成（开发者 80%）；平台提供完整 SDK 与沙箱环境。

## 不可随意变更的核心架构规则
1. 核心平台仅负责：
   - 用户认证
   - RBAC 权限
   - 插件生命周期管理
   - AI 模型调度（GPT / Claude / Composer）
2. 所有业务能力必须插件化实现，避免业务逻辑侵入核心基座。
3. 架构演进必须保持低耦合与可扩展，不破坏插件生态边界。

## 技术栈基线
- Web：Next.js 15 + React + TypeScript（App Router）
- Mobile：React Native + TypeScript（备选 PWA）
- Backend：FastAPI + PostgreSQL + Redis
- 插件通信：MCP 协议
- 沙箱隔离：WebAssembly

  1) 架构设计文档
  2) 核心平台代码骨架
  3) 插件模板 + 跨境电商示例插件
- 代码组织与语言规范：

  - 分层目录：`core/`、`plugins/`、`web/`

  - Markdown 文档

  - `Dockerfile` 与开发环境配置

## 主要交付物（阶段细化）
- 阶段一：完整架构设计文档（含技术架构图、插件规范、数据库设计）。
- 阶段二：核心平台代码骨架（Next.js 前端基座 + FastAPI 后端服务）。
- 阶段三：插件开发模板与跨境电商 MVP 示例（商品翻译、市场分析插件）。

## 插件机制规则
- 插件必须支持：独立开发、安装、升级、卸载。
- 生命周期钩子必须包含：
  - `onInstall()`
  - `onActivate()`
  - `onDeactivate()`
- 插件市场能力：
  - 开发者上架
  - 审核
  - 定价
  - 收益分成（开发者 80%）
- 安全与依赖：
  - 插件声明依赖版本和权限
  - 沙箱隔离
  - 仅可访问白名单 API

## AI 调度规则
- 端侧不进行重型 AI 计算。
- 所有模型推理与复杂运算统一由 FastAPI 核心服务处理。
- 对多模型调用做统一封装，至少包含：
  - 密钥托管
  - 用量统计
  - 额度控制

## 运行与维护规则（执行层）
1. 任意新需求（架构调整、代码补充、插件开发、双端适配、Bug 修复、功能迭代）都必须先校验与本记忆库一致。
2. 若出现冲突，优先遵循本记忆库；若必须调整，需先发起确认。
3. 新增需求只允许“扩展”而非“破坏”原有解耦结构。
4. 每次关键决策后，需更新记忆文档与决策日志。
5. 对用户给出的执行提示，需主动进行风险与优化评估；若存在隐患或有更优方案，必须先说明并确认后执行。
6. 协作类新要求（沟通方式、交付偏好、确认机制）属于长期规则，需主动沉淀到记忆库并持续复用。

## 沟通与协作基准（用户偏好）
- 用户无代码经验，且不懂英文；所有说明默认使用中文与通俗表达。
- 输出内容优先给“结论 + 下一步操作”，避免不必要术语。
- 涉及关键改动时，先确认再执行；不影响方向的小优化可先执行后说明原因。
- 代码入库采用“半自动确认”流程：我先提供本次变更清单，用户回复“确认推送”后，再执行测试、提交、推送到 GitHub，并反馈结果。
